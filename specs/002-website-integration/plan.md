**Spec**: [spec.md](spec.md)
**Branch**: 001-agentic-pipeline-refactor (continued)

## Technical Context

- **Backend pipeline**: already implemented in `src/llmxive/` (spec 001).
- **Live site host**: GitHub Pages serves `web/` from the deployed branch at `https://context-lab.com/llmXive/`.
- **Design baseline**: `web/notes/design/{index.html, styles.css, app.js, data.js}` — port wholesale, then re-wire to real data and the new pipeline.
- **OAuth proxy**: Cloudflare Worker (free tier; ~200 lines of JS). Source committed at `infra/oauth-proxy/`.
- **Local credential store**: TOML at `~/.config/llmxive/credentials.toml` (POSIX) or `%APPDATA%\llmxive\credentials.toml` (Windows). chmod 0600 on POSIX.
- **PDF rendering**: native `<embed type="application/pdf">` first; fallback to a download link.

## Constitution Check (gates)

- **I. SSOT**: `web/data/projects.json` is the only source the site reads; status_reporter is the only writer. ✅
- **II. Verified accuracy**: Every counter is computed from on-disk truth (run-log, project state, reviews). About-page narrative cites Stage values from code. ✅
- **III. Robustness**: Playwright tests exercise real DOM + real loaded JSON. The local end-to-end test runs the actual pipeline (no mocks). ✅
- **IV. Cost-effectiveness**: Cloudflare Workers free tier (100k req/day). No paid services. Local credential storage avoids re-asking the user. ✅
- **V. Fail fast**: Missing credentials prompt immediately; missing OAuth proxy URL is a top-of-page banner; PDF render failures fall back to a download link instantly. ✅

## OAuth flow (FR-012 to FR-018)

1. User clicks **Sign in with GitHub** → JS redirects to:
   `https://github.com/login/oauth/authorize?client_id=<id>&redirect_uri=<site>&state=<csrf>&scope=public_repo`
2. GitHub redirects back to the site root with `?code=<x>&state=<csrf>`.
3. JS POSTs `{code, state}` to the Worker URL (read from `<meta name="llmxive-oauth-proxy">`).
4. Worker verifies state matches the cookie it set, then POSTs to `https://github.com/login/oauth/access_token` with the secret. Returns `{access_token, scope}`.
5. JS stores `{token, user}` in localStorage; refreshes UI to show the avatar.

## Risks

- **OAuth Worker deploy is out-of-band**: Cloudflare account creation is one-time, but the deploy itself is `wrangler deploy` and not committable from CI without secrets. Acceptable: maintainers run it manually on first deploy.
- **PDF rendering varies by browser**: We test the three majors and ship a download fallback.
- **Real local pipeline run is slow**: At ~1 min/step × ~30 steps = 30 min for 2 papers. Acceptable for the e2e test gate; mark `@slow` so it's opt-in.
- **API key in conversation**: User pasted the key in chat. We will store it locally per FR-019 and add a docs note recommending rotation post-session.
